# Approvio v0.2 — Plan

> v0.2 takes the solid sequential foundation of v0.1 and makes it production-grade
> for complex real-world flows: parallel approvals with quorum, conditional steps,
> role- and relationship-based approvers, delegation, escalation/deadlines, and
> resubmission after rejection. Every feature is additive — no v0.1 public API
> changes, no v0.1 migration edits.

---

## Vision recap

v0.1 proved the core mechanic: any model can move through a sequential, audited
approval lifecycle with two strategies and full tenant awareness. v0.2 makes that
mechanic powerful enough for real enterprise workflows — the kind where finance
and legal both need to approve in parallel, an expense over $10k triggers an extra
CFO step, approvers go on leave and need to delegate, deadlines must escalate
automatically, and rejected submissions need a clean resubmit path. After v0.2,
the package covers the vast majority of approval use-cases a Laravel SaaS will
ever encounter, without pulling in any mandatory new dependencies.

## What v0.2 must do

### In scope

1. **Parallel steps with quorum rules** — a step declared `.parallel()` activates
   all assigned approvers simultaneously. Quorum rules (`any`, `all`, `n_of_m`)
   determine when the step is considered approved. Rejection by any assignee in a
   parallel step is still immediately terminal.

2. **Conditional steps via `when()`** — a step with `.when(fn($model) => bool)` is
   evaluated when the engine would activate it. If the condition returns false, the
   step is skipped (status `skipped`) and the engine advances to the next step. The
   skipped step is recorded in the audit log. A workflow where all remaining steps
   are skipped still completes.

3. **Spatie Permission integration (optional)** — a `RoleResolver` that reads roles
   from `spatie/laravel-permission`. Opt-in: the package is NOT a hard dependency.
   If a workflow uses `->role('finance')` and `spatie/laravel-permission` is not
   installed, the engine throws a clear `MissingDependencyException` with install
   instructions. If the package IS installed, it resolves the users with the named
   role (and optional team scope).

4. **Relationship-based approvers** — a `RelationshipResolver` that follows a
   dot-notation relation chain on the approvable model, e.g.
   `->relation('user.department.manager')`. The chain is evaluated lazily at step
   activation time; the final node must be a `Model` or a `Collection<Model>`.

5. **Delegation** — an assignee on an active step can delegate their responsibility
   to a deputy. The original assignee's status becomes `delegated`; a new assignee
   row is created with `assigned_via = 'delegation'`. The delegate acts on behalf of
   the original. The audit log records the full chain.

6. **Escalation + deadlines + scheduled command** — a step can declare `.deadline(hours: N)`
   and optionally `.escalateTo(fn($model) => [...])`. The new `approvio:escalate`
   Artisan command (registered as a scheduled command) scans for overdue steps and
   either adds the escalation target as a new assignee or marks the request as
   expired if no target is configured. The `expires_at` column on the request is
   honoured: an expired request rejects all further actions.

7. **Resubmit after rejection** — `$rejectedRequest->resubmit()` (or
   `$approvable->resubmit()` targeting the latest rejected request) creates a new
   `ApprovalRequest` linked via `parent_request_id`, pre-populated with the original
   snapshot and context. The `pending_changes` from the original `DraftApproval`
   request are carried forward as the starting point. A `RequestResubmitted` event
   fires.

## Public API additions

Everything below is **additive**. No existing v0.1 signature changes.

### `Approvable` trait — new methods

```php
// Resubmit the most recent rejected request for this model.
// $changes: null = carry forward parent pending_changes; array = override.
public function resubmit(
    ?Model $requester = null,
    array  $context   = [],
    ?array $changes   = null,
): ApprovalRequest;
```

### `HasApprovalActions` trait — new methods

```php
// Delegate the caller's pending assignment to $delegateTo.
public function delegate(
    ApprovalRequest $request,
    Model           $delegateTo,
    ?string         $comment = null,
): ApprovalRequest;
```

### `Approvio` facade — new methods

```php
Approvio::delegate(
    ApprovalRequest $request,
    Model           $actor,
    Model           $delegateTo,
    ?string         $comment = null,
): ApprovalRequest

Approvio::resubmit(
    ApprovalRequest $rejectedRequest,
    ?Model          $requester = null,
    array           $context   = [],
    ?array          $changes   = null,  // null = carry forward pending_changes; array = override
): ApprovalRequest
```

### `WorkflowBuilder` — new fluent methods on a step

```php
$flow->step('name')
    ->approvers(...)
    ->parallel()            // activates all approvers simultaneously (default: quorum 'any')
    ->quorum('any')         // any one approval completes the step
    ->quorum('all')         // every assignee must approve
    ->quorum('n_of_m', 2)   // exactly N of M assignees must approve
    ->when(fn ($model) => bool)   // skip this step if condition is false
    ->deadline(hours: 48)         // escalate after 48 hours with no action
    ->escalateTo(fn ($model) => [$model->cfo]); // escalation target resolver
```

## Decisions locked in v0.2

These were open questions during planning, now resolved:

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | **Parallel rejection is always terminal** (first rejection, regardless of quorum) | `shouldStepTerminateOnRejection()` is isolated — configurable threshold is non-breaking in v0.3 |
| 2 | **Conditional steps evaluate against the live model** at activation time | Snapshot-based conditions would produce the bug "I lowered the amount but CFO step still triggered"; users can opt in via `$request->snapshot` in the closure |
| 3 | **Delegation is non-revocable** in v0.2; escape hatch is cancel + resubmit | Revocation deferred to v0.3 |
| 4 | **Escalation adds a new assignee** alongside the original (does not replace) | Audit trail — "Bob was responsible; Alice added after 48 h" — matters more than convenience |
| 5 | **`expires` is a real terminal `RequestStatus`** | A separate terminal state enables clean queries and avoids status lying about reality |
| 6 | **`resubmit($changes = null)`** — null carries forward `pending_changes`; array overrides | Matches `requestApprovalFor()` pattern; no need to expose `pending_changes` as a userland concept |
| 7 | **PHPStan ratchet**: level 7 at Sprint 4, level 8 at Sprint 8 | Ratcheting earlier creates friction during design work |

## Risks and known limitations

- **Parallel step rejection is still terminal.** One rejection on a parallel step
  terminates the request regardless of how many approvals were pending. This is
  intentional for v0.2. A configurable "require N rejections" policy is v0.3.
- **Delegation is one-level only.** A delegate cannot further delegate. Attempting
  to do so throws `DelegationException`. Nested delegation chains are v0.3.
- **Escalation target is resolved at escalation time, not at submission time.**
  If the target resolver depends on model state that changed since submission, it
  resolves against the current (live) model, not the snapshot.
- **`when()` condition is evaluated at step-activation time.** If the condition
  depends on external state (e.g., a third-party API), that state is queried live.
  Caching is the application's responsibility.
- **`spatie/laravel-permission` team scope.** If the app uses teams, the
  `RoleResolver` must be given a team ID at construction time. The default
  (no team) resolves globally, which may pull in cross-tenant users on shared-DB
  setups. Document this clearly.
- **MySQL/PostgreSQL CI matrix requires Docker.** The GitHub Actions workflow for
  Sprint 8 will use `services:` blocks, which increases CI time. Accepted tradeoff.
- **Parallel step rejection policy is fixed.** Any single rejection on a parallel
  step terminates the request regardless of quorum. A configurable "require N
  rejections" policy is deferred to v0.3. Documented in README.
- **Delegation is non-revocable in v0.2.** Once an assignee delegates, they are
  locked out for the request lifetime. The escape hatch is cancel + resubmit.
  True revocation is v0.3. Documented in README.
- **Conditional steps evaluate against the live model.** Users wanting
  snapshot-based semantics must read `$request->snapshot` explicitly inside the
  condition closure. Documented in README with an example.

## Lookahead — what v0.3/v0.4 must not be blocked on

- **v0.3: DatabaseWorkflowSource** — the `approval_workflows` table already exists.
  `CodeWorkflowSource` returns definitions just like `DatabaseWorkflowSource` will.
  v0.2 must not hardcode assumptions about where definitions come from. ✅ (already
  abstracted behind `WorkflowSource` contract)
- **v0.3: Per-tenant workflow overrides** — the engine passes `$tenant` to every
  `WorkflowSource::find()` call. v0.2 must not skip that argument. ✅
- **v0.3: "Require N rejections" on parallel steps** — don't hard-code the
  "first rejection terminates" rule deep in the engine. Isolate it in a method
  `shouldTerminateOnRejection(ApprovalRequestStep)` so v0.3 can override it.
- **v0.3: Undoing delegation** — the `delegated_to` morphs are already on the
  assignee row. v0.3 can flip status back without schema changes.
- **v0.4: Filament panel** — it will need `ApprovalRequest::with('steps.assignees.assignee')`
  to render a timeline. v0.2 must not change those relation names.
- **v0.4: Notification classes** — will hook into the new v0.2 events
  (`StepEscalated`, `StepSkipped`, `RequestDelegated`, `RequestResubmitted`).
  Event constructor signatures are frozen the moment they ship.
