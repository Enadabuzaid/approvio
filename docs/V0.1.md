# Approvio v0.1 — Plan

> The first shippable version. Goal: a useful, well-tested, well-documented
> package that real Laravel apps can adopt today, even if not every feature
> from the long-term vision is in yet.

---

## Vision recap

`enadstack/approvio` lets any Laravel app add approval workflows to any model
with a single trait. The package is open-source (MIT), headless (no UI in core),
multi-tenant-aware, and built around extension points (contracts) so the
community can plug in their own approver resolvers, tenant resolvers, and
strategies.

## What v0.1 must do

The minimum viable, lovable cut. v0.1 is **shippable** — not a preview, not a
toy. It must be safe to use in production for the use-cases it claims.

### In scope

- `Approvable` trait + `HasApprovalActions` trait, fully working.
- All 5 migrations published, with the full long-term schema (so v0.2/v0.3
  don't require breaking migrations).
- **Code-defined workflows only** via PHP classes extending `Workflow`.
- **Sequential steps only** (parallel/quorum is v0.2).
- **Direct user approvers only** via closures (role/relationship is v0.2).
- **Both strategies working:** `SoftApproval` and `DraftApproval`.
- **State machine** with guarded transitions on the request.
- **Append-only audit log** capturing actor, action, comment, IP, user-agent.
- **Events** for the full lifecycle.
- **Single-DB tenancy** via `ColumnTenantResolver`, plus `NullTenantResolver`
  for non-tenant apps. Stancl/Spatie adapters are v0.3.
- **Pest test suite** covering happy paths + key error cases.
- **GitHub Actions CI** (PHP 8.2/8.3/8.4 × Laravel 11/12).
- **PHPStan** at level 6 minimum.
- **README** with install + quickstart + first workflow walkthrough.
- **CHANGELOG, CONTRIBUTING, LICENSE.**

### Explicitly out of scope (deferred)

- Parallel steps, quorum rules → **v0.2**
- Conditional steps (`when()`) → **v0.2**
- Spatie Permission integration → **v0.2**
- Relationship-based approvers → **v0.2**
- Delegation / escalation / deadlines → **v0.2**
- DB-defined workflows → **v0.3**
- Stancl Tenancy / Spatie Multitenancy adapters → **v0.3**
- Filament panel / Inertia-Vue components → **v0.4**
- Default notification classes → **v0.4**

## Acceptance criteria for v0.1

The release is ready when **all** of the following hold:

- [ ] `composer install` works on a clean clone with PHP 8.2, 8.3, 8.4.
- [ ] `vendor/bin/pest` passes against Laravel 11 and 12.
- [ ] `vendor/bin/phpstan analyse` returns no errors.
- [ ] CI matrix (3 PHP × 2 Laravel = 6 cells) is green on `main`.
- [ ] Test coverage on `src/` is at least 80% on the engine, strategies,
      state machine, and workflow builder.
- [ ] README has a copy-pasteable quickstart that produces a working
      approval flow in a fresh Laravel app.
- [ ] CHANGELOG has a `0.1.0` section with the release date.
- [ ] Repo has: `LICENSE.md`, `README.md`, `CHANGELOG.md`, `CONTRIBUTING.md`,
      `.github/workflows/tests.yml`, `.github/workflows/static-analysis.yml`.
- [ ] At least one external user (you, on a sandbox project) has installed
      the package, run the quickstart, and confirmed it works end-to-end.

## Risks and known limitations

These are real and will be documented in the README.

- **Empty-step deadlock.** If a step's approver resolver returns zero users,
  the step activates but never completes. v0.2 will add auto-skip; v0.1 just
  documents this.
- **No retry semantics.** A rejected request cannot be re-submitted; the
  user must create a new request. v0.2 will add `resubmit()`.
- **No deadline/expiry job.** The schema supports `deadline_at` and `expires_at`,
  but no scheduler enforces them. v0.2 will ship a scheduled command.
- **`pendingApprovals()` query is not paginated.** Fine for typical loads,
  but a user with thousands of pending items will feel it. v0.2 will paginate.

## After v0.1 — the lookahead

Don't build for these now, but make sure v0.1 doesn't paint us into a corner:

- **v0.2:** Parallel steps need `quorum_rule` and `quorum_count` columns on
  `approval_request_steps` — both already in the v0.1 migration. ✅
- **v0.3:** DB-defined workflows need the `approval_workflows` table — already
  in the v0.1 migration. ✅
- **v0.3:** Tenancy adapters are pluggable through the `TenantResolver`
  contract — already a contract. ✅
- **v0.4:** UI packages depend on `ApprovalRequest::actions` being queryable
  with eager loading — works today. ✅
